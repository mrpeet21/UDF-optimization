# Решение Оптимизации функции (UDF) возврата списка заказов

Перед тем как приступить к решению задач был реализован код - генератор тестовых данных (файл `data_generate.sql`)

## Описание генерации данных:

Для тестирования был подготовлен набор данных, соответствующий условию задания: 50 000 заказов и 150 000 элементов заказов, то есть в среднем по 3 элемента на один заказ. 
Дополнительно были заполнены справочные таблицы: 50 сотрудников, 200 анализов и 4 статуса заказов. Данные были сгенерированы SQL-скриптом с использованием CTE, `ROW_NUMBER()` и системной таблицы `sys.all_objects`.
После заполнения была выполнена проверка количества строк в основных таблицах.

# Задача 1
Проанализировать скрипт функции получения списка заказов и связанные с ней объекты, перечислить выявленные недочёты и потенциальные проблемы производительности.

**Анализ исходной функции `dbo.F_WORKS_LIST`**

**1. Использование многооператорной табличной функции**

Функция `dbo.F_WORKS_LIST` возвращает результат через табличную переменную `@RESULT`. Такой подход может ухудшать производительность, так как SQL Server сложнее оценить количество строк и построить оптимальный план выполнения.

**2. Построчные вызовы пользовательских функций**

Внутри `dbo.F_WORKS_LIST` для каждой строки заказа вызываются дополнительные функции. Это приводит к построчной обработке данных: для каждого заказа отдельно считаются элементы заказа и отдельно формируется ФИО сотрудника.

**3. Двойной подсчёт элементов заказа**

Функция `F_WORKITEMS_COUNT_BY_ID_WORK` вызывается два раза: отдельно для завершённых и незавершённых элементов заказа. Из-за этого таблица `WorkItem` может просматриваться повторно для одного и того же заказа.

**4. Использование `NOT IN`**

В функции подсчёта элементов заказа используется условие `NOT IN` для исключения групповых анализов. Такой вариант может быть менее безопасным при наличии `NULL` и менее удобным для оптимизации по сравнению с `NOT EXISTS`.

**5. Получение ФИО сотрудника через отдельную функцию**

ФИО сотрудника формируется через функцию `F_EMPLOYEE_FULLNAME`. При большом количестве заказов эта функция вызывается много раз, хотя те же данные можно получить через соединение с таблицей `Employee`.

**6. Ограничение `TOP 3000` применяется снаружи функции**

Клиентский запрос получает только 3000 строк, но сама функция не знает об этом ограничении. Поэтому она может выполнять лишние вычисления до того, как внешний запрос применит `TOP`.

**7. Сортировка внутри функции**

Внутри функции используется сортировка заказов по убыванию `Id_Work`. При большом количестве строк сортировка может увеличивать стоимость выполнения запроса, особенно если перед ней уже были выполнены дополнительные вычисления.

## Вывод по задаче 1

Основная проблема исходной реализации заключается в построчной обработке данных. Вместо обработки набора заказов одним запросом функция многократно вызывает дополнительные пользовательские функции для каждой строки.

Для дальнейшей оптимизации необходимо сохранить смысл функции, но изменить способ получения результата: перейти к наборной обработке данных с использованием `JOIN`, `GROUP BY`, условной агрегации и предварительной выборки только нужных 3000 заказов.

# Задача 2

## Исходный запрос

Изначально список заказов получался с помощью функции `dbo.F_WORKS_LIST()`:

```sql
SELECT TOP (3000) *
FROM dbo.F_WORKS_LIST();
```
### Результат выполнения исходного запроса:

| Показатель       | Значение |
| ---------------- | -------: |
| Количество строк |     3000 |
| CPU time         | 19501 мс |
| Elapsed time     | 19630 мс |

## Оптимизированная функция (optimized_function.sql)

Оптимизация была выполнена за счёт изменения реализации самой функции `dbo.F_WORKS_LIST()`. Структура базы данных при этом не изменялась: новые таблицы, поля и индексы не создавались.

**Что было изменено в функции**

В исходной реализации функция `dbo.F_WORKS_LIST()` для каждой строки заказа вызывала дополнительные пользовательские функции:

- `dbo.F_WORKITEMS_COUNT_BY_ID_WORK(Works.Id_Work, 0)` — подсчёт незавершённых элементов заказа;
- `dbo.F_WORKITEMS_COUNT_BY_ID_WORK(Works.Id_Work, 1)` — подсчёт завершённых элементов заказа;
- `dbo.F_EMPLOYEE_FULLNAME(Works.Id_Employee)` — получение ФИО сотрудника.

Такой подход приводил к построчной обработке данных. При большом количестве заказов функция выполняла много повторных обращений к таблицам WorkItem, Analiz и Employee.

В оптимизированной версии логика функции была переписана в наборном стиле. Сначала внутри функции выбираются только последние 3 000 заказов, затем для них одним запросом рассчитывается количество завершённых и незавершённых элементов заказа.

**Основная идея оптимизации**

В оптимизированной функции были использованы два CTE:

`TopWorks` — выбирает только 3 000 нужных заказов из таблицы `Works`;
`WorkItemCounts` — считает количество завершённых и незавершённых элементов заказа через `GROUP BY`.

Ключевая часть оптимизации:

```sql
;WITH TopWorks AS
(
    SELECT TOP (3000)
        w.Id_Work,
        w.CREATE_Date,
        w.MaterialNumber,
        w.IS_Complit,
        w.FIO,
        w.Id_Employee,
        w.StatusId,
        w.Print_Date,
        w.SendToClientDate,
        w.SendToDoctorDate,
        w.SendToOrgDate,
        w.SendToFax
    FROM dbo.Works w
    WHERE w.IS_DEL <> 1
    ORDER BY w.Id_Work DESC
),
WorkItemCounts AS
(
    SELECT
        wi.Id_Work,
        SUM(CASE WHEN wi.Is_Complit = 0 THEN 1 ELSE 0 END) AS WorkItemsNotComplit,
        SUM(CASE WHEN wi.Is_Complit = 1 THEN 1 ELSE 0 END) AS WorkItemsComplit
    FROM dbo.WorkItem wi
    INNER JOIN TopWorks tw
        ON tw.Id_Work = wi.Id_Work
    WHERE NOT EXISTS
    (
        SELECT 1
        FROM dbo.Analiz a
        WHERE a.ID_ANALIZ = wi.ID_ANALIZ
          AND a.IS_GROUP = 1
    )
    GROUP BY wi.Id_Work
)
```
После этого данные из `TopWorks` соединяются с рассчитанными значениями `WorkItemCounts`, таблицей `Employee` и таблицей `WorkStatus`.

**Таблица изменений**

| Было в исходной функции                                                     | Стало в оптимизированной функции                            | Зачем изменено                                                                        |
| --------------------------------------------------------------------------- | ----------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| `TOP (3000)` применялся во внешнем запросе после вызова функции             | `TOP (3000)` применяется внутри функции в CTE `TopWorks`    | Чтобы сначала ограничить набор заказов и не выполнять лишние вычисления               |
| Для каждого заказа дважды вызывалась функция `F_WORKITEMS_COUNT_BY_ID_WORK` | Подсчёт выполняется одним запросом через `GROUP BY`         | Чтобы убрать повторные обращения к `WorkItem`                                         |
| Завершённые и незавершённые элементы считались отдельными вызовами функции  | Оба значения считаются через `SUM(CASE WHEN ...)`           | Чтобы получить оба счётчика за один проход по данным                                  |
| ФИО сотрудника получалось через функцию `F_EMPLOYEE_FULLNAME`               | Данные сотрудника берутся через `LEFT JOIN Employee`        | Чтобы убрать построчный вызов функции                                                 |
| Использовался `NOT IN` для исключения групповых анализов                    | Используется `NOT EXISTS`                                   | Более безопасная и удобная для оптимизации проверка                                   |
| Таблица `WorkStatus` соединялась с полным набором заказов внутри функции    | `WorkStatus` соединяется только с выбранными 3 000 заказами | Чтобы уменьшить объём обрабатываемых данных                                           |
| Обработка была построчной                                                   | Обработка стала наборной                                    | Чтобы SQL Server обрабатывал сразу набор строк, а не выполнял много отдельных вызовов |

## Вывод по 2 задаче

Оптимизация была выполнена внутри функции `dbo.F_WORKS_LIST()`. Основное изменение заключалось в отказе от построчных вызовов пользовательских функций и переходе к наборной обработке данных.

Вместо многократного вызова `F_WORKITEMS_COUNT_BY_ID_WORK()` для каждого заказа подсчёт элементов теперь выполняется одним группирующим запросом. Вместо вызова `F_EMPLOYEE_FULLNAME()` используется соединение с таблицей Employee. Также `NOT IN` был заменён на `NOT EXISTS`.

Таким образом, функция стала выполнять меньше повторной работы, а внешний способ её вызова остался прежним. Структура базы данных не изменялась, индексы не создавались.
